# Routine: AI事例記事 自動生成パイプライン

> このファイルは **定期実行エージェント(Routine / cron)が毎回読んで実行する手順書**。
> cron 側のプロンプトは短くてよい:「`docs/ROUTINE_記事自動生成.md` を読み、1イテレーション実行せよ」。
> 設計の前提は `PROJECT_GOAL.md`(単一の真実の源)に従う。矛盾したら PROJECT_GOAL を優先。
> 最終更新: 2026-06-24

---

## 0. このRoutineのゴールとNGライン

- **ゴール**: 1イテレーションで「一次情報で裏取りした事例記事1本(日英)＋ヒーロー画像」を**下書きとして**用意する。
- **最重要ルール(PROJECT_GOAL §7 運用ルール)**:
  - 各事例は **必ず一次情報を1つ以上**当たってから書く。
  - 元ニュースレターの**焼き直しは禁止**(Googleの information gain 評価で沈む＋著作権リスク)。自分の言葉・独自フォーマットで書く。
  - **出典を明記**する(後述の `sources` で一次/二次を区別)。
- **NG(やってはいけない)**:
  - 出典の取れない数字・固有名詞を書く(「たぶん」「と言われている」で埋めない。不明なら未設定にする)。
  - 実在人物の顔写真・他社アプリのスクショを取得して自サイトにホストする(著作権・肖像権)。画像は §4 の方法のみ。
  - 本番への自動 push(§6。既定は下書き→人間が承認)。

---

## 1. 1イテレーションの流れ(概要)

```
STEP0 次に書く事例を選ぶ
STEP1 一次情報をゴリゴリ探す(リサーチ・プロトコル)
STEP2 事実の裏取り・正規化(数字/固有名詞)
STEP3 記事ファイルを書く(content/cases/<slug>.ts, スキーマ準拠)
STEP4 ヒーロー画像を作る(検証つき)
STEP5 ビルド検証(npm run build)
STEP6 下書きとして提出(ブランチ+要約。人間が承認→マージで公開)
```

---

## STEP0: 次に書く事例を選ぶ

1. `PROJECT_GOAL.md` §8「最初に投入する10件」の表を見る。
2. `content/cases/` に**まだ無い** slug を上から順に1件選ぶ(既存: coconote / pep-ai / once / jungle / cal-ai)。
3. 10件を書き終えていたら、新規追加はせず「候補が尽きた」と報告して終了(勝手に質の低い事例を足さない)。

選定基準(PROJECT_GOAL §7): ①日本語情報が薄い ②日本人が気になる ③数字を裏取りできる。

---

## STEP1: 一次情報をゴリゴリ探す(リサーチ・プロトコル)

**「探す順番」= 一次情報から。** 二次情報(報道・まとめ)は補強と相互チェックにのみ使う。

### 一次情報(primary)= まずこれを取りに行く
- 創業者**本人**の発信: X(旧Twitter)/ LinkedIn / 個人ブログ / Threads。買収・収益・節目の**本人投稿**は最強の一次情報。
- **公式プロダクト**: アプリ公式サイト / 料金ページ(価格は必ず公式で確認) / App Store・Google Play のリスティング(DL数・レビュー数・公開日)。
- **本人が語った一次素材**: 本人出演のポッドキャスト/YouTubeインタビュー、本人の Product Hunt ローンチ投稿、登壇スライド。
- **公的・構造化データ**: 会社の公式プレスリリース、Crunchbase の資金調達、(あれば)決算・買収の公式発表。

### 探索テクニック(ゴリゴリ探す具体策)
- WebSearch を**角度を変えて複数回**投げる: 「<人物名> founder」「<アプリ名> revenue ARR」「<アプリ名> acquired」「<人物名> interview podcast」「<アプリ名> pricing」「<人物名> X twitter」。
- 固有名詞の**表記揺れ・別人混同**を必ず潰す(例: 共同創業者が複数いる/同名の別プロダクトがある)。最低2ソースで人物・役割を一致確認する。
- 数字は**一次→二次の順**で確認し、食い違ったら**一次を採用**し、食い違いの事実をコード内コメントに残す。
- 取れた各 URL は「これは一次か二次か」「何が確認できたか」をメモしながら進める(そのまま STEP3 の `sources` になる)。

### 撤退条件
- 一次情報が**1つも**見つからない/数字がどうしても裏取りできない場合は、**その事例は書かずスキップ**し、次の候補へ。理由を報告に残す。

---

## STEP2: 事実の裏取り・正規化

- **収益 `mrrUsd`(USD建てMRRに正規化)が主ソートキー**。換算ルール:
  - MRR表記 → そのまま。 ARR → `÷12`。 年商/累計(total) → 実態に合わせ慎重に(基本は ARR 換算できる時のみ)。 日次(daily) → `×30`。
  - `revenueType` に元の種別("mrr"|"total"|"arr"|"daily")を入れる。元の生表記は `revenueRaw`(日英)に残す。
- **タグは固定辞書(enum)のキーのみ**。自由入力禁止。使えるキーは `content/types.ts` の `CATEGORIES` / `TECH` / `MARKETING` を参照(無いものは足さない/最も近いものを選ぶ。techが不明なら空配列でよい)。
- 不明な数値フィールド(users / periodMonths 等)は**捏造せず未設定**にする。

---

## STEP3: 記事ファイルを書く(スキーマ準拠)

- 出力先: `content/cases/<slug>.ts`。ひな型は既存の `content/cases/cal-ai.ts` を手本にする(最も新しく完全な見本)。
- `content/types.ts` の `Case` 型を満たす。`satisfies Case` を付ける。
- ファイル冒頭に**典拠コメント**(どの一次/二次情報で何を確認したか、数値の換算メモ)を必ず書く。
- 翻訳対象(日英 両方必須): `title` / `summary` / `revenueRaw` / `painPoint` / `body.free` / `body.premium` / `playbook` / `failureNote` / `sources[].title`。
- `body.free` は概要・背景・プロダクト・痛点。`playbook` は再現手順3〜7ステップ。
- **`body.premium` は“完全版”を書く(最重要)**。見出しだけ・「(※要点のみ)」「実装時はここに入れる」等の**プレースホルダで終わらせない**。各見出しの下に、裏取りした事実・数字・解釈を伴う本文を書き切る(目安: 見出し4〜6本＋各2〜5文、日英とも)。途中で止めるのは未完成として扱い、提出しない。
- **出典 `sources`(必須・明記)**: 取れた URL を `{ url, title:{ja,en}, kind:"primary"|"secondary" }` で列挙。**primary を最低1つ**含めること。primary が0なら STEP1 の撤退条件に戻る。
- **本人Xポストは“status URL”で入れる(写真も自動で付く)**: `sources` の primary に **`https://x.com/<user>/status/<id>` 形式**(特定ツイート)を入れると、`src/components/SourceTweets.tsx` が自動で2つ表示する:
  1. 記事本文へのツイート**公式埋め込み**(`SourceTweets`)。
  2. タイトル下の**本人ポートレート**(`FounderPortrait` — そのツイート投稿者のXアバターを埋め込みで表示。自サーバに再ホストしない)。
  どちらも **primary な status URL が1つも無ければ自動で非表示**(= 本人写真は「あれば出す/無ければ無し」)。プロフィールURL(`x.com/<user>`)では出ないので、買収告知・収益公表・節目など**具体的な1ツイートのstatus URL**を優先的に拾う。著作権セーフ。
- 最後に **`content/cases/index.ts` に import 追加＋ `cases` 配列へ追記**(これを忘れるとページが生成されない)。

---

## STEP4: ヒーロー画像(next/og で完全自動・作業ゼロ)

**画像は next/og のコード生成カードに一本化済み(2026-06-24 ユーザー決定)。** 著作権リスクゼロ・日本語フォント不要・ja/en共通で、**この STEP では基本的に何もしなくてよい**。

- 実装は共有レンダラ `src/lib/og-card.tsx`。事例の**構造化データ**から複数バリアントのカードを自動描画する:
  - `hero`(収益見出し) … `src/app/[locale]/cases/[slug]/opengraph-image.tsx` 経由で **OGP/Twitter＋記事冒頭ヒーロー**に使用。
  - `stack`(集客チャネル＋技術スタック) … `/api/card?slug=<slug>&v=stack` 経由で **記事本文中の2枚目の画像**として自動表示。
- 記事ページがこれらを自動表示するので、**1記事あたり画像が2枚以上**自動で付く。`heroImage` の手動設定は不要。
- Routine がやることは「STEP3 で構造化データ(`appName` / `mrrUsd`＋`revenueType` / `periodMonths` / `users` / `personName` / `category` / `techStack` / `marketingTags`)を正しく埋める」ことだけ。特に **`marketingTags`/`techStack` を充実させるほど2枚目の `stack` カードが充実する**(=写真が増える)。
- 画像をさらに増やしたい/種類を足したいときは `src/lib/og-card.tsx` に新バリアントを追加する(全記事に一括反映)。
- カードのテキストは**英数字主体**(例 `$50M ARR · 18 mo · 15M users`)。これは satori の既定フォントで日英共通に描画するための設計なので、**カードに日本語を足さない**(足すと豆腐文字になる)。
- 画像の見た目を変えたいときは `opengraph-image.tsx` を編集する(個別記事ごとの画像作業は不要)。
- 例外: どうしても固有のキービジュアルを使いたい特別な記事だけ、`heroImage: "/images/<slug>.png"` を手動設定すれば上書きできる(任意・通常は使わない)。

---

## STEP5: ビルド検証

- `npm run build` を実行し、**型エラー0・全ページ生成**を確認する。
- 失敗したら原因(多くは辞書キー違い/Localized 欠け/index.ts 追記漏れ)を直してから次へ。緑になるまで提出しない。

---

## STEP6: 下書きとして提出(既定 = 人間が承認して公開)

PROJECT_GOAL の品質ルール上、**既定は「下書き→ユーザー承認→公開」**。**`main` に直接 push/マージはしない。**
実行環境によって提出方法が変わる(どちらも「下書き」):

### A. クラウド実行(GitHub上で動く / PC offでも動く既定の運用)
1. 新規ブランチを切る: `content/<slug>`(例 `content/halo-ai`)。
2. 変更(`content/cases/<slug>.ts`, `content/cases/index.ts` など)を commit。
3. ブランチを push し、**`main` 宛ての Draft PR(下書きPR)を作成**する(`gh pr create --draft --base main` 等)。PRタイトルは事例名、本文に下記「報告」を書く。
4. **PRはマージしない**(ユーザーがレビュー→マージ→Vercel自動デプロイで本番 aidevcases.com へ)。

### B. ローカル実行(手元PCで動かす場合)
1. 新規ブランチ `content/<slug>` を切って commit する(`main` に直接 commit しない)。
2. push はせず、ローカルブランチのまま「報告」を出す。ユーザーが確認してから push/マージする。

### 報告(共通)
事例名 / 確認した一次情報URL(特にprimary) / 主要数字とその出典 / ビルド結果 / 提出先(PRのURL または ブランチ名)。スキップ・候補切れの場合はその理由。

> 切替: 完全自動公開にしたい場合のみ、A-3/4 を「`main` へ直接マージ(またはpush)」に変更する(ビルド緑が絶対条件)。ただし §0 の品質ルールを満たせる確証がある事例に限る。既定は Draft PR。

## 守るべき制約(提出前チェックリスト)

- [ ] 一次情報(primary)を最低1つ `sources` に明記した
- [ ] すべての数字に出典がある(不明な数値は未設定にした)
- [ ] 固有名詞・共同創業者・役割を最低2ソースで一致確認した
- [ ] タグは固定辞書のキーのみ使った
- [ ] 日英すべての翻訳対象フィールドを埋めた
- [ ] `index.ts` に追記した
- [ ] ヒーロー画像は next/og 自動生成(個別画像作業は不要 / カードに日本語を足していない)
- [ ] `npm run build` が緑
- [ ] `main` に直接 push していない(既定運用の場合)

## スケジュール(cron)登録メモ

- 登録は `schedule` スキル(またはユーザーが対話で)で行う。cron プロンプト例:
  > 「`docs/ROUTINE_記事自動生成.md` を読み、未投入の事例を1件、手順どおりに下書き作成して報告して」
- 頻度の目安(PROJECT_GOAL の10件リストを消化する想定):
  - 慎重運用: 週2〜3件(裏取りに余裕)。
  - 標準: 1日1件(10日で10件完了)。
- まずは**手動で1回**走らせて品質を確認してから cron 化する(2026-06-24 ユーザー方針)。
